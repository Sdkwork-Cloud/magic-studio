# 01. 产品设计与需求范围

## 1. 产品定位

Magic Studio V2 的核心定位不是“一个 AI 视频生成器”，而是“面向创作者的视频创作工作台”。它同时覆盖：

- 灵感输入：文本、图片、音频、脚本、素材导入
- 智能生成：图像、视频、音乐、音效、语音、角色、提示词增强
- 结构化创作：`canvas` 节点编排、`film` 剧本式创作、`magic-cut` 时间线剪辑
- 工程管理：工作区、项目、资产、笔记、云盘、浏览器、发布与分发
- 平台扩展：模型接入、技能市场、插件市场、桌面能力扩展

对标的不是单一竞品，而是“剪映 + Runway + Descript + ComfyUI + 桌面创作 IDE”的组合能力。

## 2. 核心用户

| 用户类型 | 主要目标 | 关键诉求 |
| --- | --- | --- |
| 短视频创作者 | 快速生成和剪辑可发布内容 | 模板快、预览快、导出稳、资产复用 |
| 影视化创作者 | 从脚本到分镜到视频生成 | 角色、场景、道具、镜头、连续性管理 |
| AI 工作流创作者 | 可视化编排复杂生成流程 | 节点编排、参数复用、可调试、可复用 |
| 内容团队 / 工作室 | 团队项目与资产管理 | 工作区、项目分层、存储治理、发布协同 |
| 高级创作者 / 技术型用户 | 深度控制生成和工程 | 本地能力、插件、脚本、外部工具集成 |

## 3. 需求问题定义

当前视频创作工具通常存在以下问题：

- AI 生成工具强，但工程管理弱，结果难以沉淀为可维护项目。
- 剪辑工具强，但多模态生成和智能编排能力不足。
- 节点式工作流强，但对普通创作者门槛高，缺少产品级创作入口。
- 云端服务强，但本地素材、大文件、私有化、安全与性能受限。

Magic Studio V2 要解决的是：

1. 如何把“灵感输入”快速转成“可编辑工程”。
2. 如何让不同创作模式共享资产、项目和生成上下文。
3. 如何既支持轻量创作，也支持专业工程级创作。
4. 如何在本地优先与云能力之间取得平衡。

## 4. 需求范围

### 4.1 已纳入的核心能力范围

| 能力域 | 当前入口 | 业务目标 |
| --- | --- | --- |
| 统一创作门户 | `/portal` | 统一用户起点，承接跨模态内容生成与跳转 |
| 图片创作 | `/image` | 文生图、图生图、风格化、素材入库 |
| 视频创作 | `/video` | 文生视频、图生视频、风格迁移、延长、口型同步 |
| 音乐 / 音效 / 语音 / 音频 | `/music` `/sfx` `/voice` `/audio` | 声音生成、配音、语音实验、音频处理 |
| 角色创作 | `/character` | 角色设定、角色资源管理、角色驱动生成 |
| 影视创作模式 | `/film` `/film/editor` | 从剧本、角色、场景到镜头和生成的结构化创作 |
| 剪辑创作模式 | `/magic-cut` | 时间轴剪辑、素材编排、预览、属性编辑、导出 |
| 画布创作模式 | `/canvas` | 节点式可视化编排、自由布局、分组、连接 |
| 对话与提示词 | `/chat` `/prompt` | 自由对话编辑、提示词增强、创意辅助 |
| 工程与资产 | `/editor` `/assets` `/drive` `/notes` | 项目、资产、笔记、文件协同管理 |
| 平台扩展 | `/portal/skills` `/portal/plugins` | 技能与插件扩展体系 |
| 账号与商业 | `/login` `/profile` `/vip` `/task-market` | 用户、会员、任务市场、下载入口 |

### 4.2 非功能需求

本产品不是普通内容站点，必须从一开始把非功能需求定义成一级需求：

- 大素材可处理：支持长视频、大音频、高分辨率图片、复杂时间线工程
- 本地优先：即便网络不稳定，也应保持工程读取、编辑和大部分素材处理可用
- 多模式一致性：不同模式下的资产、项目、设置、账户体验要一致
- 安全可信：本地文件、账号令牌、模型密钥、插件执行必须可治理
- 易扩展：模型、插件、工作流、创作模式可持续扩展
- 易维护：模块边界清晰，变更不牵连全局

## 5. 当前实现与需求映射

### 5.1 当前产品优势

- 创作模式丰富，已不是单页应用，而是完整工作台。
- 路由和包结构基本与能力域对齐，后续扩展成本相对可控。
- 统一门户、资产中心、工作区服务已经形成产品底座。
- 桌面端能力较强，适合承载专业视频创作场景。

### 5.2 当前缺口

| 需求方向 | 当前缺口 | 影响 |
| --- | --- | --- |
| 工程闭环 | 项目同步与发布仍是 mock | 难以形成真正的项目交付闭环 |
| 插件生态 | 插件市场存在，但运行时未完成 | 扩展能力无法形成平台护城河 |
| AI 治理 | 后端 SDK 与直连模型并存 | 成本、审计、安全和结果一致性难统一 |
| 安全模型 | 本地文件权限过宽 | 与专业桌面应用安全要求不匹配 |
| 跨模式统一性 | `canvas`、`film`、`magic-cut` 的统一工程协议尚不完整 | 资产复用和模式切换成本较高 |

## 6. 目标产品标准

面向行业领先水平，Magic Studio V2 的目标产品标准应是：

1. 一个入口承接所有创作意图，但不牺牲专业模式深度。
2. 所有生成结果默认进入工程和资产体系，而不是成为散落文件。
3. 任一创作模式都可回到统一的项目、资产、历史和发布体系。
4. AI 不只是“生成按钮”，而是可参与创意理解、编辑、优化、组织和发布的编排能力。
5. 平台支持个人创作者，也支持团队工作室和潜在私有化场景。

## 7. 范围边界

为保持高内聚，以下内容不应直接混入当前主干范围：

- 与视频创作无关的通用社交平台能力
- 无边界的插件执行能力
- 页面级临时直连第三方服务且不纳入统一鉴权治理的实现
- 不经过资产中心和项目体系的“旁路式”素材落地

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 需求覆盖完整度 | 用户从灵感输入到导出发布的链路完整 | `L3-L4` | 打通发布、插件、跨模式工程闭环 |
| 场景清晰度 | 每个模式有明确用户和任务边界 | `L4` | 继续收敛模式边界与入口策略 |
| 一致性 | 资产、项目、AI、设置在全产品统一 | `L3` | 统一项目图谱与 AI 接入策略 |
| 专业深度 | 至少一个模式达到专业级深度 | `L3` | 强化 `magic-cut` 和 `film` 深度能力 |
| 平台扩展性 | 新模型、新插件、新工作流接入成本低 | `L2-L3` | 完成插件运行时和扩展协议治理 |

## 9. 结论

从需求范围看，Magic Studio V2 已具备成为“多模态视频创作平台”的产品基础，而不是单点工具。下一阶段的关键，不是继续平铺页面，而是围绕工程闭环、统一引擎协议、插件生态和安全治理，把现有广度转化为真正的平台深度。
