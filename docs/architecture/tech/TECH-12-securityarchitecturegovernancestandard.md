# 12. 安全架构与治理标准

## 1. 安全目标

Magic Studio V2 作为桌面优先创作平台，安全风险同时来自：

- 本地文件访问
- 外部网络访问
- 账号与会话
- AI 模型密钥与调用
- 插件与扩展
- 发布与更新

安全目标应明确为：

- 最小权限
- 默认隔离
- 敏感数据受保护
- 所有高风险行为可审计
- 安全规则可自动化验证

## 2. 主要威胁模型

| 威胁面 | 说明 |
| --- | --- |
| 本地文件滥用 | 过宽的文件系统权限可能导致误删、越权读取、恶意插件利用 |
| 命令执行风险 | 本地媒体处理、终端、系统命令能力被滥用 |
| 插件扩展风险 | 插件若无沙箱，可能获得过强本地能力 |
| 会话和令牌泄露 | 登录态、模型密钥、第三方平台密钥若存储不当会泄露 |
| AI 内容风险 | 提示词注入、不安全输出、敏感资产泄露 |
| 供应链风险 | 多依赖、多脚本、多平台发布流程带来的依赖污染风险 |

## 3. 当前安全现状

### 3.1 优势

- 桌面能力已经有 service / policy 架构雏形。
- 业务接入后端多数走统一 SDK，而不是完全散落手写请求。
- 已存在一些工程治理和验证脚本，说明具备纳入门禁的基础。

### 3.2 关键风险

当前最突出的风险是桌面能力默认授权过宽：

- `fs:scope` 允许 `**`
- 文件读写、删除、重命名等能力默认广泛开放
- 多项 webview / window 能力默认开启

此外还存在：

- 插件运行时尚未建立真正的权限模型
- LLM / 存储 / 媒体平台配置中存在密钥型数据，需要更严格的本地保护策略
- AI 直连供应商路径会增加密钥暴露和审计难度

## 4. 安全架构标准

### 4.1 身份与会话

要求：

- 统一会话管理
- 访问令牌、刷新令牌与本地缓存策略清晰
- 桌面端敏感令牌优先使用受保护存储
- 多环境和多模式下会话行为一致

### 4.2 文件与路径安全

要求：

1. 所有路径操作必须经过白名单校验。
2. 写、删、重命名必须区分风险级别。
3. 用户工作区、缓存、导出、插件、日志目录必须隔离。
4. 不允许业务模块自行拼任意系统路径。

### 4.3 命令与执行安全

要求：

- 媒体命令参数必须受控构造
- 终端与外部命令默认不开给普通业务流
- 插件执行环境与主进程隔离
- 高风险命令进入审计日志

### 4.4 插件安全

插件未来必须具备：

- capability manifest
- 权限申请
- 沙箱目录
- 生命周期隔离
- 网络访问策略
- 版本兼容策略
- 审计与熔断

### 4.5 AI 安全

要求：

- 模型接入有供应商白名单
- 敏感内容与不安全输出有策略处理
- 外部模型调用避免直接暴露高价值密钥
- 生成与编辑行为可审计、可回放

## 5. 数据保护要求

| 数据类型 | 安全要求 |
| --- | --- |
| 登录态 token | 受保护存储、最小暴露 |
| 模型密钥 | 不进入普通日志，不明文散落 |
| 存储平台密钥 | 支持服务端代理模式，避免客户端永久保存高权限密钥 |
| 素材文件 | 限定作用域、限制路径、具备删除与恢复策略 |
| 插件数据 | 与主工作区隔离 |

## 6. 更新与发布安全

桌面端更新与发布应满足：

- 安装包来源可信
- Release 产物可校验
- Updater 来源可信且可配置
- 版本切换有回滚策略
- 多平台构建链路受控

## 7. 当前优先级最高的安全整改项

1. 将 Tauri 默认文件系统权限收敛到工作区白名单。
2. 为插件系统设计 capability manifest 与沙箱目录。
3. 为本地保存的 LLM / 存储等密钥建立更强保护策略。
4. 收敛 AI 直连路径，避免敏感密钥散落前端。
5. 将高风险命令和文件操作纳入审计日志。

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 最小权限 | 默认拒绝，高风险按需开放 | `L2` | 收缩 capability 与路径白名单 |
| 数据保护 | 敏感配置安全存储、最小暴露 | `L2-L3` | 加强本地机密管理 |
| 插件安全 | 插件沙箱、权限、审计闭环 | `L1-L2` | 完成插件运行时安全设计 |
| AI 安全治理 | 模型、内容、密钥、成本都可控 | `L2-L3` | 统一 AI Gateway 和策略层 |
| 可审计性 | 高风险行为可追踪、可回放 | `L2` | 建立统一审计日志 |

## 9. 结论

Magic Studio V2 当前最大的安全问题，不是“有没有安全意识”，而是“强能力已经具备，但最小权限治理尚未完全跟上”。行业领先应用必须把安全规则前置到架构层，而不是等到上线后补洞。
