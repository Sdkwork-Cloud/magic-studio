# 11. 性能架构与优化标准

## 1. 性能目标

视频创作应用的性能目标与普通 Web 页面完全不同。Magic Studio V2 的性能目标应覆盖：

- 启动速度
- 大工程加载
- 大素材导入
- 复杂时间线交互
- 节点画布交互
- 预览与导出
- 长任务执行

目标不是“能打开”，而是“在专业工作流中可持续使用”。

## 2. 当前性能基础

从现有实现看，系统已经具备一些正确的性能方向：

- 路由与页面按包懒加载
- 启动阶段采用 deferred task，避免首屏阻塞
- 资产中心延迟初始化
- 桌面端将媒体重处理下沉到本地工具链
- `canvas` 已使用 Quadtree 裁剪等画布性能策略
- 本地优先目录和工具包减少了对远端网络的强依赖

这些基础说明系统具备继续优化到专业级的可能。

## 3. 关键性能场景

### 3.1 启动性能

关键指标：

- 应用冷启动时间
- 首屏可交互时间
- 首屏布局稳定性
- 启动后延迟初始化对用户操作的影响

### 3.2 素材性能

关键指标：

- 大文件导入耗时
- 素材缩略图生成速度
- 元数据分析耗时
- 大批量素材扫描与查询耗时

### 3.3 引擎性能

| 引擎 | 关键指标 |
| --- | --- |
| `canvas` | 平移、缩放、拖拽、框选、连线时的帧率与节点规模上限 |
| `film` | 脚本结构、镜头列表、AI Copilot 交互与预览切换延迟 |
| `magic-cut` | 时间线滚动、拖拽、预览、波形、导出准备耗时 |

### 3.4 任务性能

关键指标：

- 生成任务提交响应
- 轮询与状态同步效率
- 本地媒体任务吞吐
- 导出任务稳定性
- 大任务取消响应时间

## 4. 建议的性能预算

以下预算是面向专业桌面创作平台的推荐值：

| 指标 | 推荐目标 |
| --- | --- |
| 桌面冷启动到主界面可交互 | `<= 3s` |
| 常规路由切换 | `<= 300ms` 视觉完成 |
| 10k 节点以内画布操作 | 保持交互流畅，核心操作接近 `60 FPS` |
| 中型时间线拖动与缩放 | 不出现明显卡顿，核心操作 `>= 30 FPS` |
| 单素材导入反馈 | `<= 500ms` 给出明确状态 |
| 大文件媒体分析 | 有异步反馈，不阻塞主线程 |
| 长任务取消反馈 | `<= 1s` 给出取消中状态 |

这些指标可以按设备分层，但必须有明确预算，而不是凭感觉优化。

## 5. 当前潜在瓶颈

| 瓶颈方向 | 说明 |
| --- | --- |
| 包体增长 | 功能广度很大，若缺乏预算控制，首屏和模式切换都会变重 |
| 引擎状态规模 | 时间线、节点图、影视结构复杂后，React 渲染压力会显著增加 |
| 素材处理链路 | 大量缩略图、波形、转码、分析会放大 CPU / IO 压力 |
| AI 状态更新 | 轮询任务如果无统一调度，会造成无效请求和 UI 抖动 |
| 本地索引与缓存 | 若缺少统一缓存淘汰和索引优化，大工程会逐步变慢 |

## 6. 性能优化标准

### 6.1 前端优化标准

- 重型路由必须懒加载
- 非关键初始化必须延迟执行
- 引擎页面需要视窗化、分区渲染和局部更新
- 复杂状态变更需避免全树重渲染
- 资源卡片、列表、素材面板应支持分页或虚拟化

### 6.2 引擎优化标准

- `canvas`：节点裁剪、局部连线重算、缩放级别差异化渲染
- `film`：列表与预览分离、重结构数据增量更新
- `magic-cut`：时间线分段渲染、预览缓存、波形与缩略图异步化

### 6.3 本地任务优化标准

- 所有 FFmpeg / 大文件处理走异步任务系统
- 输出进度和阶段信息
- 支持取消与资源回收
- 缓存目录和正式资产目录分离

## 7. 观测与告警

性能架构若无观测就无法真正管理。建议建立以下指标：

| 指标域 | 代表指标 |
| --- | --- |
| 启动 | cold start、interactive、deferred init latency |
| 路由 | 路由切换耗时、chunk 大小 |
| 素材 | 导入耗时、缩略图耗时、分析耗时 |
| 引擎 | 交互帧率、渲染耗时、内存峰值 |
| AI | 提交延迟、成功率、轮询耗时、结果落地耗时 |
| 本地任务 | queue 深度、失败率、取消耗时、平均执行时长 |

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 启动性能 | 首屏快且延迟初始化不打断操作 | `L3-L4` | 建立明确启动预算与监控 |
| 引擎交互性能 | 复杂画布和时间线仍可流畅操作 | `L3` | 强化渲染分区和虚拟化 |
| 本地处理性能 | 大任务高吞吐、可取消、可恢复 | `L3-L4` | 增强任务治理和缓存策略 |
| 观测能力 | 性能问题可定位、可回归验证 | `L2-L3` | 建立统一指标体系 |
| 性能门禁 | 构建和发布存在性能预算门禁 | `L3` | 扩大预算校验范围 |

## 9. 结论

Magic Studio V2 已经具备性能架构的正确基础，但尚未完全进入“预算化治理”阶段。若要对标行业领先应用，必须把性能目标从经验判断转成明确预算、观测指标和发布门禁。
