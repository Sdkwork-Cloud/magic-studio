# 10. 技术选型与工程治理

## 1. 技术选型目标

Magic Studio V2 的技术选型必须服务于两个核心前提：

- 这是一个长期演进的桌面优先创作平台
- 它既要支持快速前端迭代，也要承载本地重能力和工程治理

因此技术选型不应只追求“开发快”，还必须兼顾：

- 稳定性
- 性能
- 可扩展性
- 边界治理
- 跨平台交付

## 2. 当前技术栈

| 层次 | 当前选型 | 选择理由 |
| --- | --- | --- |
| 前端框架 | `React + TypeScript` | 生态成熟，适合复杂交互和大型前端工程 |
| 构建工具 | `Vite` | 开发体验好，适合多包和现代前端 |
| Monorepo | `pnpm workspace + Turbo` | 适合包级拆分、缓存与增量构建 |
| 桌面框架 | `Tauri 2` | 轻量、跨平台、可接 Rust 本地能力 |
| 本地服务 | `Rust` | 性能强、类型安全、适合本地能力服务化 |
| UI 基础 | `Radix UI + Tailwind` | 适合搭建可控、可复用组件体系 |
| 编辑器能力 | `Monaco`、`TipTap`、`xterm` 等 | 满足 IDE / 富文本 / 终端场景 |
| SDK 接入 | `retired generic app SDK` + `core-pc-react` | 统一后端访问与会话治理 |
| 测试 | `Vitest` + Node 脚本测试 | 适合前端与工程脚本混合验证 |

## 3. 当前选型优点

### 3.1 前端层面

- React 适合复杂页面装配与工作台化界面。
- TypeScript 对大规模包结构和领域模型很重要。
- Vite 在开发和多模式构建上较灵活。

### 3.2 桌面层面

- Tauri 适合跨平台桌面发布。
- Rust service 化设计非常契合本地工具链、任务执行与安全控制。

### 3.3 工程治理层面

- Monorepo 允许将能力拆到 package 级别。
- 生成 SDK 方向降低了手写 API 漂移风险。
- 已存在 service encapsulation 审计脚本，说明治理意识较强。

## 4. 当前需要注意的选型代价

| 选型 | 风险 / 代价 |
| --- | --- |
| React + 多引擎页面 | 状态和性能治理难度高 |
| Monorepo | 若边界不清，容易形成隐性依赖网 |
| Tauri + Rust | 需要较强原生与前端协同治理能力 |
| 双 AI 路径 | 会削弱 SDK-first 的技术标准 |
| 多创作模式并行 | 若无统一协议，会造成技术债快速累积 |

## 5. 工程治理标准

### 5.1 代码组织治理

建议继续坚持：

- 领域包优先，不按技术层随意散文件
- 公共能力进入 shared package
- 引擎级逻辑独立，不与页面壳混杂
- 生成能力统一经 service / facade 暴露

### 5.2 依赖治理

应建立以下规则：

1. 业务包不得直接新增手写业务 API 客户端。
2. 新本地能力必须经 runtime facade 暴露。
3. 新创作模式必须说明其项目、资产、任务、导出归属。
4. 新共享组件必须证明复用价值，避免共享层污染。

### 5.3 配置与模式治理

当前工程已存在多种模式与验证脚本：

- `development`
- `test`
- `staging`
- `production`
- `git-sdk`

这对产品化很重要，但也要求：

- 模式差异必须可验证
- 样式和构建行为不能在不同模式漂移
- SDK 模式切换不能导致行为不一致

## 6. 当前工程治理优势

- 构建、验证、发布脚本较完整。
- 已有 bundle 自包含、CSS 预算、SDK 模式一致性、嵌入资源等检查。
- 已有 service boundary 审计脚本。
- GitHub Actions 已支持多平台发布产物。

## 7. 仍需强化的治理能力

| 治理项 | 当前不足 |
| --- | --- |
| 架构 review gate | 对新增旁路能力仍需更强门禁 |
| 安全门禁 | 本地权限和插件能力缺少强制性检查 |
| 引擎协议治理 | `canvas` / `film` / `magic-cut` 缺少统一变更标准 |
| 发布治理 | 工程发布链路尚未完全产品化 |
| 依赖透明度 | AI 直连和旁路实现需继续收敛 |

## 8. 行业领先工程标准

行业领先的创作平台工程体系，至少应具备：

1. 架构规则可脚本校验，而不是仅靠口头约定。
2. 安全、性能、包体、模式一致性都进入 CI gate。
3. 每个新模块都必须说明边界、依赖和验收标准。
4. 跨端行为差异可回归验证。
5. 发布流程可追踪、可回滚、可审计。

## 9. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 技术栈匹配度 | 选型与产品目标强一致 | `L4` | 继续坚持桌面优先与平台化方向 |
| 工程治理力度 | 规则可落地、可校验 | `L3-L4` | 将更多架构规则纳入自动化门禁 |
| 模式一致性 | 多环境、多 SDK 模式行为稳定 | `L3-L4` | 扩大验证覆盖面 |
| 跨端治理 | Web / Desktop 共用逻辑清晰 | `L4` | 继续限制平台差异泄漏 |
| 扩展性 | 新能力接入成本低且可控 | `L3` | 优先补齐插件和引擎扩展协议 |

## 10. 结论

当前技术选型总体正确，特别适合构建桌面优先的多模态创作平台。后续真正需要加强的不是替换主技术栈，而是把现有技术优势转化为更严格的工程治理和演进纪律。
