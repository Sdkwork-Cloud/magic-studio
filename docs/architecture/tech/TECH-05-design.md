# 05. 数据模型与资产-工程存储设计

## 1. 设计目标

作为视频创作应用，Magic Studio V2 的数据与存储设计必须服务以下事实：

- 素材体积大、类型多、生命周期长
- 同一素材会在生成、影视创作、画布、剪辑中被反复引用
- 工程不是单文件，而是项目、资源、执行历史、配置和导出物的组合
- 桌面优先意味着本地存储不是缓存，而是主生产环境

因此设计目标应为：

- 统一标识
- 统一引用
- 本地优先
- 可同步
- 可恢复
- 可审计

## 2. 核心数据对象

### 2.1 工程对象

| 对象 | 说明 | 当前实现基础 |
| --- | --- | --- |

| `ProjectType` | 工程类型，如 `VIDEO` `FILM` `CANVAS` `CUT` | 已有类型映射 |

### 2.2 资产对象

## 3. 存储体系

### 3.1 当前存储策略

当前已经存在较清晰的本地优先设计：

- 默认素材策略为 `local-first-sync`
- 支持 `local-only`、`server-only`
- 默认本地根目录为 `~/.sdkwork/magicstudio`
- 资产支持浏览器 VFS、Tauri 文件系统、远端 URL、混合模式
- 桌面工具包支持本地目录、文件系统、SQLite 与媒体处理能力

### 3.2 推荐的分层存储模型

```text
User Workspace Root
  /workspaces
  /projects
  /assets
  /cache
  /exports
  /db
  /logs
  /temp
```

不同类型数据建议的持久化位置：

| 数据类型 | 推荐主存储 | 说明 |
| --- | --- | --- |
| 项目元数据 | SQLite / structured JSON | 支持查询、过滤、恢复 |
| 资产索引 | SQLite 或索引仓库 | 统一检索和引用治理 |
| 原始素材 | 本地文件系统 / 对象存储 | 体积大，适合文件化 |
| 中间产物 | cache / temp | 可回收，不应与原始资产混存 |
| 导出物 | exports | 与工程运行数据隔离 |
| 用户设置 | local storage / tauri store / config file | 轻量配置 |
| 聊天记录 | 本地文件 + 索引 | 当前已具备类似形态 |

## 4. 资产中心设计标准

资产中心是本系统最关键的数据底座，应满足：

1. 所有资产都有稳定 `assetId` / `uuid`。
2. 所有引擎引用资产时优先使用 `assetId`，而不是裸路径。
3. 资产具备作用域、标签、收藏、状态、版本信息、引用关系。
4. 资产定位与存储解耦，业务只拿 locator，不处理底层路径策略。
5. 资产导入时自动补充媒体分析和元数据。

当前实现已经满足其中较大部分，这是全系统的重要优势。

## 5. 工程与资产的关系

建议统一约束如下：

| 关系 | 规则 |
| --- | --- |
| 项目引用资产 | 只能通过资产引用关系绑定 |
| 生成结果入库 | 先成为资产，再绑定到项目或引擎对象 |
| 画布节点引用 | 保存 `assetId + derived metadata`，不保存临时裸路径 |
| 时间线资源引用 | 优先按 `assetId` 解析，避免文件移动后失效 |
| 影视资源引用 | 角色、场景、道具、分镜都应能绑定资产和生成结果 |

## 6. 当前差距

| 差距 | 说明 | 影响 |
| --- | --- | --- |
| 项目发布对象缺失 | 发布信息与分发目标尚未形成稳定实体 | 无法完整支持交付和回滚 |
| 引擎统一项目协议不足 | `canvas` / `film` / `magic-cut` 共用工程模型仍待加强 | 难以实现跨模式复用 |
| 版本治理不足 | 项目快照、资产版本、导出版本治理仍可加强 | 恢复和对比能力有限 |
| 数据恢复标准不足 | 大工程崩溃恢复与断点恢复机制需加强 | 专业创作场景下风险较高 |

## 7. 行业领先的数据设计标准

要达到行业领先水平，应进一步具备：

1. 项目快照与增量历史。
2. 素材引用图与反向引用分析。
3. 缓存与正式资产的明确生命周期。
4. 本地工程恢复、损坏检测与自动修复。
5. 团队模式下可扩展到共享存储与协同索引。

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 统一数据模型 | 项目、资产、任务、引擎对象可统一引用 | `L3-L4` | 建立统一项目图谱 |
| 存储策略清晰度 | 原始、缓存、导出、索引分层明确 | `L4` | 强化目录标准和生命周期治理 |
| 本地优先能力 | 断网下核心流程仍可运行 | `L4` | 补齐恢复和同步策略 |
| 版本与恢复 | 可回滚、可恢复、可审计 | `L2-L3` | 建立快照和修复机制 |
| 跨模式复用 | 素材和结果可在多模式复用 | `L3-L4` | 强化资产引用协议 |

## 9. 结论

Magic Studio V2 在数据与存储方向的基础是正确的，尤其资产中心和本地优先策略具备明显平台潜力。下一步的重点是把“资产统一”继续升级为“工程统一”，让所有创作模式真正共享同一套项目与数据治理主干。
