# 08. AI接入与多模态生成架构

## 1. 设计目标

Magic Studio V2 的 AI 架构不应停留在“页面里调用模型”层面，而应服务于完整创作体系：

- 多模态生成
- 提示词优化
- 对话式编辑
- 影视创作辅助
- 节点工作流执行
- 结果归档与复用

目标标准是：

- 模型接入统一
- 任务编排统一
- 结果归档统一
- 成本与权限可治理
- 可扩展到多模型、多供应商、多业务模式

## 2. 当前 AI 接入现状

### 2.1 已有主路径

当前主要路径是：

- 通过 `retired generic app SDK` 接入后端生成能力
- 通过 `core-pc-react` runtime 处理会话与客户端能力
- 各业务 service 负责封装具体生成行为

这条路径的优点是：

- 能统一鉴权
- 便于后端治理
- 更适合生产环境

### 2.2 仍存在的并行路径

当前也存在直连模型路径，例如：

- `genAIService` 中存在基于 `@google/genai` 的直连能力
- `chat` / `prompt` 等部分能力存在本地直连或降级行为

这种混合路径短期提升了可用性，但长期存在明显问题：

- 模型策略不一致
- 审计与成本统计困难
- 安全边界和密钥存储复杂
- 行为一致性难保证

## 3. 当前多模态生成能力

### 3.1 覆盖范围

当前多模态生成已覆盖：

- 图片生成
- 视频生成
- 图生视频
- 视频风格迁移
- 视频延长
- 口型同步
- 语音 / 音频生成
- 提示词增强
- 聊天式生成辅助

### 3.2 统一任务特征

从当前实现可以看到，系统已经逐步向统一任务模型靠拢：

- 请求带模型、场景、比例、时长、参考素材
- 结果包含主资源、资源列表、封面、时长、元数据
- 异步轮询状态
- 成功后转为资产或输出结果

这说明“统一生成执行模型”已经有基础，只需进一步制度化。

## 4. 目标 AI 架构

### 4.1 推荐分层

```text
Intent Layer
  Portal / Chat / Prompt / Film / Canvas / MagicCut

AI Orchestration Layer
  Task Router
  Prompt Strategy
  Reference Asset Builder
  Safety & Policy
  Cost & Quota
  Result Post-Processor

Provider Access Layer
  Backend AI Gateway
  Model adapters
  Fallback strategies

Persistence Layer
  Generation Task
  Asset Center
  Project Binding
  Audit Log
```

### 4.2 统一策略

建议明确：

1. 生产路径以后端 AI Gateway 为主。
2. 直连模型仅作为受控降级或本地实验能力。
3. 所有生成结果必须进入资产中心。
4. 所有任务必须记录模型、版本、参数、成本、来源和状态。
5. AI 输出应可进入项目、时间线、画布、影视结构，而不是停留在页面文本结果。

## 5. AI 与创作引擎的关系

### 5.1 Portal

Portal 是意图入口，适合负责：

- 收集用户目标
- 选择模式
- 汇总参考素材
- 创建生成会话

### 5.2 Film

Film 中 AI 主要服务：

- 剧本理解
- 角色与场景生成
- 分镜建议
- 镜头内容生成
- 连续性维护

### 5.3 MagicCut

MagicCut 中 AI 应服务：

- 素材粗剪建议
- 字幕与文案辅助
- 时间线结构优化
- 节奏 / 转场建议
- 自动生成封面和模板

### 5.4 Canvas

Canvas 中 AI 应服务：

- 节点执行
- 参数推荐
- 结果比较
- 工作流模板化
- 复杂链路的自动装配

## 6. 安全与治理要求

AI 架构必须自带治理，而不是事后补规则：

| 治理项 | 要求 |
| --- | --- |
| 模型接入 | 后端登记、版本可控、能力可配置 |
| 密钥管理 | 页面不保存高风险供应商密钥 |
| 内容安全 | 输入输出具备策略校验与风险标记 |
| 成本控制 | 任务维度可统计 token / 调用成本 |
| 审计追踪 | 能追溯是谁、在何项目、用何模型生成了什么 |

## 7. 当前差距

| 差距 | 说明 |
| --- | --- |
| 双路径并存 | SDK 路径与直连路径未完全统一 |
| 统一代理能力不足 | Chat 结果还未普遍转化为工程动作 |
| 结果治理不足 | 模型输出的质量评分、比对、回放标准可进一步完善 |
| AI 观测不足 | 成本、时延、成功率缺少统一平台视角 |

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 接入统一性 | 所有模型通过统一策略接入 | `L3` | 收敛为 AI Gateway 优先 |
| 多模态能力广度 | 图像、视频、音频、角色、文本协同 | `L4` | 继续统一结果协议 |
| 任务治理 | 每次生成有状态、成本、审计、恢复 | `L3` | 建立统一任务治理中心 |
| 与工程整合 | AI 结果可自然进入项目和引擎 | `L3-L4` | 强化动作化和归档标准 |
| 安全与成本治理 | 密钥、配额、内容风险都可控 | `L2-L3` | 后端集中治理 |

## 9. 结论

Magic Studio V2 的 AI 能力广度已经具备平台潜力，当前真正需要补的不是“再接一个模型”，而是把已有能力收敛成统一的 AI 编排和治理体系。只有这样，AI 才会从页面功能升级为整个平台的生产力核心。
