# 09. canvas / film / magiccut 创作引擎设计

## 1. 设计目标

`canvas`、`film`、`magic-cut` 是 Magic Studio V2 最具差异化价值的三类创作引擎：

- `canvas` 代表自由编排和高级工作流
- `film` 代表结构化影视创作
- `magic-cut` 代表时间线剪辑与成片生产

三者不应被看作三个无关页面，而应被定义为共享底座、各自擅长不同创作任务的三种引擎。

## 2. 引擎共性抽象

### 2.1 必须统一的能力

| 共性能力 | 说明 |
| --- | --- |
| 项目上下文 | 三种引擎都必须运行在项目中 |
| 资产引用 | 三种引擎都必须通过统一资产协议读写素材 |
| 执行历史 | 用户动作、生成记录、导入记录需可追溯 |
| 选择与焦点模型 | 当前选中对象、焦点区域、属性编辑必须有统一思想 |
| 撤销重做 | 至少核心编辑能力应支持历史回退 |
| 导出与发布 | 输出需要映射到统一导出或发布链路 |

### 2.2 推荐统一工程模型

```text
Project Graph
  -> Asset References
  -> Engine Document
      -> CanvasDocument
      -> FilmDocument
      -> MagicCutTimeline
  -> Generation Executions
  -> Export Records
  -> Publish Records
```

这个模型的价值是：引擎文档可不同，但项目图谱一致。

## 3. Canvas 引擎设计

### 3.1 当前特点

Canvas 已具备：

- 节点拖拽
- 连接关系
- 分组
- 迷你地图
- 缩放与对齐
- 框选
- 上下文菜单
- 抽屉式辅助面板
- Quadtree 裁剪等性能设计

### 3.2 目标定位

Canvas 应定位为：

- 高级工作流编排器
- AI 工作流实验与模板平台
- 多模态资产加工管线的可视化入口

而不是简单的“看起来很炫的自由画板”。

### 3.3 设计重点

- 节点类型标准化
- 输入输出端口协议
- 执行状态与日志
- 模板复用与分享
- 节点权限与插件扩展

## 4. Film 引擎设计

### 4.1 当前特点

Film 编辑器已具备：

- 左侧项目导航
- 中央工作区
- 右侧 AI Copilot
- 预览模式
- 剧本、角色、道具、场景、分镜、生成等分栏

### 4.2 目标定位

Film 应定位为：

- 面向叙事型视频的结构化创作引擎
- 管理“角色、地点、道具、镜头、场次、连续性”的专业工作台

### 4.3 设计重点

- 剧本到分镜的结构化映射
- 角色 / 场景 / 道具资产化
- 镜头生成任务绑定
- 连续性规则
- 可进入后续剪辑和导出链路

## 5. MagicCut 引擎设计

### 5.1 当前特点

MagicCut 已具备典型的剪辑工作区：

- 资源面板
- 播放器
- 时间线
- 属性区
- 导出
- 模板保存
- 键盘快捷键
- 事件总线

### 5.2 目标定位

MagicCut 应定位为：

- 类剪映的专业剪辑工作台
- AI 与传统剪辑融合的最后一公里生产引擎

### 5.3 设计重点

- 时间线数据结构标准化
- 轨道与片段的引用一致性
- 预览性能与缓存策略
- 非破坏式编辑
- 效果链、转场、模板与导出治理

## 6. 三引擎协作关系

### 6.1 推荐流转路径

| 起点 | 可流转目标 | 说明 |
| --- | --- | --- |
| Portal | Film / MagicCut / Canvas | 从轻量入口进入专业模式 |
| Film | MagicCut | 影视结构生成后进入精剪 |
| Canvas | Film / MagicCut / Asset Center | 工作流结果沉淀为项目内容 |
| Chat / Prompt | 任意引擎 | 对话式编辑应成为跨引擎辅助层 |

### 6.2 必须避免的情况

- 三个引擎各自产生独立项目体系
- 三个引擎各自维护不同素材引用协议
- 三个引擎无法共用 AI 执行历史
- 导出和发布在三个引擎中各写一套逻辑

## 7. 当前差距

| 差距 | 影响 |
| --- | --- |
| 统一项目图谱尚未完全形成 | 三引擎联动成本较高 |
| 撤销重做和历史机制未统一表达 | 维护与协作复杂度增加 |
| 引擎间转换协议不足 | 难以形成真正的平台级生产链路 |
| 插件化引擎扩展尚未建立 | 长期扩展性受限 |

## 8. 行业领先引擎标准

若要达到行业领先水平，三引擎应满足：

1. 引擎间可共享资产、结果、历史和导出链路。
2. 每个引擎有自己的专业深度，但不割裂主平台。
3. AI 能嵌入每个引擎的核心操作，而不仅是外围辅助。
4. 引擎文档具备快照、恢复和版本化能力。
5. 引擎可被插件扩展，但扩展边界受控。

## 9. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 引擎专业深度 | 三个引擎至少各自具备清晰专业价值 | `L3-L4` | 继续深化各自核心模型 |
| 主干统一性 | 引擎都挂接统一项目与资产体系 | `L3` | 建立统一项目图谱 |
| 流转顺畅度 | 模式间转换低损耗 | `L3` | 定义互转协议 |
| AI 融合度 | AI 深入编辑动作而非外围辅助 | `L3` | 建立动作层和代理层 |
| 可扩展性 | 引擎可被能力和插件扩展 | `L2-L3` | 设计引擎扩展点 |

## 10. 结论

`canvas`、`film`、`magic-cut` 是 Magic Studio V2 走向行业领先的关键抓手。真正的架构目标不是让三者“都存在”，而是让三者在统一工程和资产底座上形成完整的创作引擎矩阵。
