# 14. 行业对标、评估标准与演进路线

## 1. 对标方法

Magic Studio V2 不适合只对标一个产品，而应按能力域对标行业最优：

| 能力域 | 建议对标对象 | 对标重点 |
| --- | --- | --- |
| 剪辑体验 | 剪映 / CapCut | 时间线交互、模板、导出稳定性、消费级易用性 |
| AI 视频生成 | Runway | 模型编排、生成质量、创作流整合 |
| 文本 / 语音驱动编辑 | Descript | 文本驱动编辑、语音处理、内容重构 |
| 节点工作流 | ComfyUI | 工作流表达力、节点生态、可调试性 |
| 平台工作台体验 | Figma 类产品方法论 | 工作台一致性、组件化、平台扩展性 |

结论是：

> Magic Studio V2 的方向应是“把行业头部产品的强项整合进同一平台主干”，而不是只复制某一个点状能力。

## 2. 当前综合评分

基于本目录各章判断，当前综合成熟度可概括为：

| 维度 | 当前等级 | 目标等级 | 说明 |
| --- | --- | --- | --- |
| 产品范围 | `L4` | `L5` | 覆盖面强，但仍需闭环深化 |
| 总体架构 | `L4` | `L5` | 底座较稳，仍需统一引擎和治理层 |
| AI 架构 | `L3` | `L5` | 广度好，治理和统一性需加强 |
| 创作引擎 | `L3-L4` | `L5` | 已有差异化，但跨引擎主干仍待完善 |
| 资产与工程 | `L4` | `L5` | 方向正确，是最值得继续强化的底座 |
| 插件生态 | `L2` | `L5` | 目前与领先平台差距最大 |
| 性能 | `L3` | `L5` | 需进入预算化治理阶段 |
| 安全 | `L2-L3` | `L5` | 桌面强能力需要配套强治理 |
| 质量与发布 | `L3-L4` | `L5` | 自动化基础较好，需补全链路验收 |

## 3. 行业领先标准清单

若要进入行业领先区间，Magic Studio V2 至少应满足以下 10 条硬标准：

1. 统一项目图谱，三大创作引擎共享工程底座。
2. 统一资产协议，所有结果可追踪、可复用、可回收。
3. AI 接入统一治理，不再长期并存多套旁路策略。
4. 插件具备真正的权限、沙箱、生命周期和扩展协议。
5. 时间线、画布、影视结构都具备专业级可维护数据模型。
6. 本地能力默认最小权限，路径和命令可治理。
7. 性能有预算、有指标、有门禁。
8. 发布有多平台标准化流程、可回滚能力和更新治理。
9. 关键创作链路有端到端验收标准。
10. 产品入口轻量，专业模式足够深，模式切换低损耗。

## 4. 差距分析

### 4.1 与剪映类产品的差距

- 剪辑生产链路和模板生态仍需深化
- 导出稳定性、效果链、模板沉淀体系还可加强

### 4.2 与 Runway 类产品的差距

- AI 模型治理、结果评分、创作编排的统一性仍不足
- 生成结果与项目结构的深度整合还有空间

### 4.3 与 Descript 类产品的差距

- 文本驱动编辑还未全面变成可执行工程动作
- 音频 / 语音 / 脚本编辑与成片生产的闭环有待增强

### 4.4 与 ComfyUI 类产品的差距

- 节点生态与插件扩展仍在早期
- 工作流模板、调试与社区复用能力仍可显著加强

## 5. 演进路线

本章保留战略阶段与目标优先级，真正用于派工、并行实施、门禁验收的详细 step 见：

- [15-架构落地步骤-并行实施计划](./15-架构落地步骤-并行实施计划.md)

### 5.1 第一阶段：平台主干收敛

目标：

- 收敛 AI 接入
- 完成统一项目图谱
- 收紧桌面权限
- 让发布链路真正产品化

关键动作：

1. 用统一 AI Gateway 替代业务层长期直连模型路径。
2. 让 `canvas`、`film`、`magic-cut` 都挂接统一工程协议。
3. 替换 mock 的同步与发布逻辑。
4. 建立路径白名单和命令白名单。

### 5.2 第二阶段：创作引擎深化

目标：

- 提升引擎专业深度
- 强化跨模式流转
- 建立性能预算和恢复能力

关键动作：

1. 时间线、画布、影视结构统一历史与快照机制。
2. 强化预览、缓存、导出和大工程恢复。
3. 推动 AI 从辅助结果升级为可执行编辑动作。

### 5.3 第三阶段：平台生态化

目标：

- 形成插件生态
- 建立模板和工作流生态
- 支持团队和企业级治理

关键动作：

1. 完成插件 manifest、沙箱和权限协议。
2. 建立工作流模板与分享机制。
3. 支持更强的团队项目、存储和安全治理能力。

## 6. 管理层可用的评估表

以下评估表可直接用于季度架构评审：

| 维度 | 问题 | 通过标准 |
| --- | --- | --- |
| 产品 | 是否形成从输入到发布的完整闭环 | 至少主链路完全闭环 |
| 架构 | 是否新增了旁路实现 | 不允许新增绕过 SDK / runtime / asset center 的旁路 |
| 引擎 | 三大引擎是否共享主干模型 | 共享项目、资产、执行历史 |
| 安全 | 是否仍存在明显过宽权限 | 发布版必须最小权限 |
| 性能 | 是否有预算和监控 | 关键指标必须可观测 |
| 质量 | 是否存在端到端回归缺口 | 关键链路必须可验证 |
| 发布 | 是否支持稳定多平台交付 | 三大平台构建和升级稳定 |
| 生态 | 插件是否具备沙箱和权限模型 | 没有则不算平台级插件体系 |

## 7. 最终判断

Magic Studio V2 当前已经站在一个很有潜力的位置上：它不是从零开始构思的平台，而是已经拥有广泛能力和相对成熟底座的平台雏形。

如果后续继续按本目录的标准推进，重点补齐以下四件事：

1. 统一项目图谱与三引擎主干。
2. 统一 AI 治理与任务治理。
3. 完成插件运行时与安全体系。
4. 将性能、安全、质量纳入硬门禁。

那么它有机会从“功能很多的创作工具集合”进化为“行业领先的视频创作平台”。

## 8. 2026-04-07 Step 01 优先级重排结论

结合本轮 `Step 01` 现实审计，当前执行优先级进一步冻结为以下顺序：

### 8.1 P0 级

1. 冻结共享主干实施边界，避免在 `main` 脏工作区中直接抢改共享文件。
2. 推进 `Step 02`，统一 `ProjectGraph` 与工程协议，并收敛 `projectService.ts` 的主路径责任边界。
3. 推进 `Step 03`，把资产引用和目录标准固定为后续引擎统一依赖。
4. 推进 `Step 04`，收敛 AI 主路径，限制 `@google/genai` 直连只作为受控 fallback。
5. 推进 `Step 05`，完成 capability 收敛与桌面最小权限闭环。

### 8.2 P1 级

1. 在 `Step 05` 完成后推进 `Step 10`，把插件从页面壳升级为权限与沙箱受控运行时。
2. 在 `Step 02` 与 `Step 05` 完成后推进 `Step 11`，移除 `projectService.ts` 的交付 mock，打通同步、导出、发布链路。

### 8.3 P2 级

1. `Step 06-09` 在 `Step 02-04` 共享协议冻结后进入最小闭环实施。
2. `Step 12-13` 在主链路真实可跑后再进入预算门禁与总验收，不允许提前口头放行。

### 8.4 当前阶段判断

当前阶段最重要的不是新增页面或继续扩展功能广度，而是严格执行：

- 先共享主干
- 后创作引擎
- 先最小权限
- 后插件与交付
- 先真实主链路
- 后性能与最终发布

这是当前从 `internal` 向更高发布阶段推进的唯一正确路径。
