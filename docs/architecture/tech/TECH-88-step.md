# Step能力兑现证明包模板

## 1. 文档定位

一个 step 只有完成“代码修改”还不够，必须同时完成“能力兑现证明”。  
本文件定义任一 step 宣告完成时，最少必须提交哪些证据，才能证明：

- 该 step 对应的架构能力已真实落地
- 该能力不是一次性演示，而是可重复验证
- 该能力不是局部优化，而是已经可供下游 step 使用

## 2. 统一原则

任一 step 的证明包必须满足以下原则：

1. 可复验：任何人拿到证明包都能按命令、路径、步骤重新验证。
2. 可追溯：能追溯到具体代码目录、文档条目、测试命令和验收结论。
3. 可回滚：能说明如果本 step 回退，哪些能力会失效、如何恢复。
4. 可交接：下游 lane 能基于证明包判断是否允许继续推进。
5. 可裁决：架构 owner、QA owner、发布 owner 能据此给出 Go / No-Go。
6. 可审计：必须能关联到 `docs/review/` 中的对应评审与回写记录。

## 3. 推荐目录结构

推荐每个 step 的证明包按如下结构沉淀到交付物或 PR 附件中：

```text
docs/step/evidence/step-XX/
  00-summary.md
  01-design-freeze.md
  02-code-map.md
  03-test-report.md
  04-verification-report.md
  05-risk-and-rollback.md
  06-architecture-writeback.md
  07-handoff.md
```

如果当前迭代暂不在仓库中创建实体目录，则至少需要在 step 执行记录、PR 描述或交接单中完整覆盖上述八类信息。

## 4. 统一证明包模板

### 4.1 `00-summary.md`

必须说明：

- 当前 step 编号与名称
- 对应架构能力编号或文档章节
- 本次实现范围与明确非目标
- 当前结论：`通过 / 有条件通过 / 不通过`
- 对应 `docs/review/` 证据路径

### 4.2 `01-design-freeze.md`

必须说明：

- 当前冻结的接口、协议、目录边界
- 本次设计与架构文档对齐的章节
- 本次没有解决但被明确延期的问题

### 4.3 `02-code-map.md`

必须说明：

- 本次修改的核心目录与文件
- 哪些共享文件被触达，owner 是否批准
- 哪些 mock、fallback、兼容层被移除、降级或保留

### 4.4 `03-test-report.md`

必须说明：

- 自动化测试命令
- 本地构建或脚本验证命令
- 覆盖的场景边界
- 未覆盖的风险点

### 4.5 `04-verification-report.md`

必须说明：

- 主链路验证步骤
- 预期结果与实际结果
- 至少一个可重复演示路径
- 至少一个失败/拒绝路径验证

### 4.6 `05-risk-and-rollback.md`

必须说明：

- 当前剩余风险
- 风险等级
- 回滚方式
- 回滚后会影响的下游 step

### 4.7 `06-architecture-writeback.md`

必须说明：

- 回写了哪些架构文档或 step 文档
- 哪些能力已经兑现
- 哪些兼容层仍然保留

### 4.8 `07-handoff.md`

必须说明：

- 解锁的下游 step
- 下游应遵守的边界
- 不允许下游做的事情
- 交接人、接收人、交接时间

## 5. Step 级最小证明要求

| Step | 最少必须证明的内容 |
| --- | --- |
| `00` | 门禁、命名、检查点、执行纪律已经统一冻结 |
| `01` | 差距矩阵、共享文件 owner、优先级和风险已经冻结 |
| `02` | Film / MagicCut / Canvas 共享同一项目主干，不再新增私有项目模型 |
| `03` | 关键资源统一以 `assetId` 为主引用，资产中心成为唯一落地主干 |
| `04` | AI 请求通过统一任务中心流转，fallback 为受控兼容层而非平行主路径 |
| `05` | capability 与命令进入最小权限和策略校验，拒绝路径有效 |
| `06` | Film 形成“脚本 -> 分镜 -> 生成 -> 预览 -> 回写”的真实链路 |
| `07` | MagicCut 形成“导入 -> 编辑 -> 导出 -> 恢复”的真实链路 |
| `08` | Canvas 形成“节点 -> 执行 -> 结果归档 -> 下游输出”的真实链路 |
| `09` | Chat / Prompt 可以驱动至少一条真实工程动作链路，而非只输出建议文本 |
| `10` | 插件具备 manifest、权限、沙箱、生命周期与拒绝路径 |
| `11` | 同步、导出、安装、发布至少一条真实交付链路可跑通 |
| `12` | 性能具备预算、观测、门禁三件套，且指标来自真实链路 |
| `13` | 可以用统一证据证明架构文档中的核心能力已实现，并给出 Go / No-Go |

## 6. 一票否决项

出现以下任一情况，证明包直接判定为无效：

1. 只有截图或演示视频，没有可重复命令与验证路径。
2. 只有测试通过记录，没有能力闭环说明。
3. 共享文件被改动，但没有 owner 批准和移交说明。
4. 主路径仍然依赖 mock，却没有明确降级为兼容层。
5. 没有回滚说明，却宣告 step 已完成。
6. 没有架构回写，却宣告能力已兑现。

## 7. 推荐评审顺序

每个 step 结束时，建议按以下顺序审查证明包：

1. Lane owner 先做自审，确认八类证据完整。
2. 集成 owner 审查共享文件、交接和合流风险。
3. QA owner 审查测试、验证路径和失败路径。
4. 架构 owner 审查能力是否真正对齐架构目标。
5. 对涉及安全、性能、发布的 step，再由专项 owner 做最终复核。

## 8. 结论

如果没有证明包，step 就只能叫“实现已提交”，不能叫“能力已兑现”。  
本文件的目的，是把所有 step 从“看起来做完了”提升为“有证据证明真的做完了”。
