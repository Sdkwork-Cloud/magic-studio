# 13. 测试质量、安装部署、发布运维

## 1. 质量目标

Magic Studio V2 需要的不是单一测试体系，而是“研发质量体系”。其范围应包含：

- 单元和组件测试
- 服务与协议测试
- 引擎关键路径测试
- 桌面能力验证
- 构建与发布验证
- 安装包与更新验证

## 2. 当前测试与验证基础

当前工程已经具备较好的质量基础：

- `Vitest` 测试入口
- Node 侧脚本测试入口
- service encapsulation policy 校验
- bundle 自包含校验
- 模式与样式一致性校验
- SDK 模式一致性校验
- Tauri 嵌入资源校验
- Release API 目标校验

当前说明系统已开始把“架构规则”和“发布规则”纳入自动化，而不是只测 UI。

## 3. 建议的测试分层

### 3.1 单元测试

适用范围：

- 领域函数
- 数据转换
- 状态映射
- 任务状态机
- 资产模型处理

### 3.2 组件与页面测试

适用范围：

- 资源列表
- 属性面板
- 工作区切换
- 关键表单与导入流程

### 3.3 服务与协议测试

适用范围：

- SDK facade
- asset center
- workspace service
- AI 请求与结果映射
- runtime facade

### 3.4 引擎回归测试

适用范围：

- `canvas` 节点操作
- `film` 结构编辑
- `magic-cut` 时间线编辑
- 大文件导入与导出任务

### 3.5 桌面与发布验证

适用范围：

- Tauri build
- bundle 结构
- 更新资源
- 多平台安装包
- 权限配置与嵌入资源

## 4. 安装与开发标准

### 4.1 本地开发前置条件

- Node.js
- `pnpm`
- Rust toolchain
- Tauri 依赖环境

### 4.2 关键开发命令

```bash
pnpm install
pnpm dev
pnpm tauri:dev
pnpm build
pnpm tauri:build
pnpm tauri:bundle
pnpm test
pnpm typecheck
pnpm lint
```

### 4.3 关键验证命令

```bash
pnpm verify:release:artifacts
pnpm verify:sdk-mode-parity
pnpm verify:mode-style-parity
pnpm verify:service-standards
pnpm audit:services:policy
```

这些命令的价值在于把“看起来能跑”升级成“按标准通过”。

## 5. 部署与发布设计

### 5.1 当前发布形态

当前仓库已具备 GitHub Actions 发布流水线，支持：

- Windows `NSIS`
- Ubuntu `AppImage` / `DEB`
- macOS `DMG`（Intel / Apple Silicon）

发布流程包括：

1. 创建或更新 GitHub Release
2. 多平台构建桌面安装包
3. 上传 release assets

### 5.2 当前更新机制

运行时更新仍依赖：

- `https://api.sdkwork.com/app/v3/api/app/update/latest?app=magic-studio`

说明桌面安装包发布与应用更新元数据并不是同一套系统，需要继续保持版本对齐和发布纪律。

## 6. 推荐的发布门禁

每次正式发布前，至少应通过以下门禁：

| 门禁项 | 说明 |
| --- | --- |
| 类型检查 | 防止构建前的接口和类型漂移 |
| 单测 / 服务测试 | 保证核心逻辑未回归 |
| service boundary 审计 | 防止架构旁路扩散 |
| 模式一致性验证 | 防止不同模式行为漂移 |
| 桌面 bundle 校验 | 防止安装包缺资源或结构异常 |
| 权限与安全检查 | 防止危险配置进入发布版 |
| 多平台冒烟验证 | 至少覆盖主平台安装与启动 |

## 7. 运维与观测

虽然本产品是桌面优先应用，但仍需要运维体系：

- 构建产物可追踪
- 更新失败可回滚
- 客户端版本分布可统计
- 关键崩溃、任务失败和更新失败可观测
- AI 任务、素材导入、导出链路具备运行指标

## 8. 当前不足

| 方向 | 当前不足 |
| --- | --- |
| 端到端验收 | 缺少面向核心创作链路的系统化验收定义 |
| 安全门禁 | 权限与敏感配置尚未形成强制发布门禁 |
| 发布闭环 | 项目同步与发布功能本身仍未完全产品化 |
| 运维指标 | 客户端和创作链路的观测指标还可更系统 |

## 9. 行业领先质量标准

行业领先应用应做到：

1. 关键创作链路有回归用例。
2. 桌面安装、启动、更新、回滚具备标准化验证。
3. 架构规则、性能预算、安全规则都进入 CI gate。
4. 发布不是一次性行为，而是可回溯、可比较、可回滚的流程。
5. 私有化、企业版、受限环境安装也有明确标准。

## 10. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 测试覆盖结构 | 单元、服务、引擎、发布验证齐全 | `L3-L4` | 增加核心创作链路验收 |
| 发布稳定性 | 多平台构建稳定、更新可控 | `L4` | 继续强化多平台冒烟和回滚策略 |
| 安装部署规范 | 安装、升级、卸载、私有化都有文档和门禁 | `L3` | 补充受限环境和企业部署标准 |
| 观测能力 | 失败、崩溃、更新、导出可观测 | `L2-L3` | 建立客户端运维指标体系 |
| 质量治理力度 | 架构、性能、安全进入发布门禁 | `L3` | 扩大自动化门禁范围 |

## 11. 结论

Magic Studio V2 已经有相当不错的发布与验证基础，但还需要从“工程可构建”走向“产品可稳定交付”。面向行业领先标准，质量治理必须覆盖创作链路、桌面链路和平台治理链路三大方向。
