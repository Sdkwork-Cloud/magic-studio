# 06. 前端架构与组件化设计

## 1. 设计目标

Magic Studio V2 的前端不是普通内容站点前端，而是桌面级创作应用前端。其核心挑战包括：

- 页面数量多、模式多、状态复杂
- 创作类页面需要长生命周期状态和高频交互
- 既要兼容 Web，又要承载桌面重能力
- 要支持复杂布局、面板、多工作区、编辑器体验

因此前端架构目标应为：

- 应用壳稳定
- 页面按领域装配
- 组件可复用
- 状态可控
- 平台能力隔离
- 引擎型页面具备长期演进能力

## 2. 当前前端架构

### 2.1 App Shell

当前应用壳由以下部分组成：

- `src/app/bootstrap.ts` 负责全局初始化
- `src/app/App.tsx` 负责根应用装配
- `src/router/routes.ts` 与 `registry.tsx` 负责路由和页面注册
- `src/layouts/*` 负责不同模式的布局骨架

当前布局已经体现出“按模式区分工作台”的思路，例如：

- `main`
- `generation`
- `creation`
- `vibe`
- `magic-cut`
- `notes`
- `blank`
- `none`

这说明系统已经不是单布局应用，而是支持多创作场景的工作台化 UI。

### 2.2 启动与延迟初始化

当前 `bootstrap` 已做了较合理的延迟初始化：

- 注入统一 platform API
- 初始化 SDK runtime
- 初始化 i18n
- 延迟注册包级 i18n
- 延迟初始化 asset center
- 延迟采集环境信息
- 桌面端延迟检查更新

这是桌面创作应用应当具备的正确方向，因为它避免把所有重初始化压在首屏。

## 3. 组件化分层标准

建议将前端组件统一分为以下层次：

### 3.1 Shell 组件

职责：

- 应用框架
- 布局
- 路由容器
- 全局导航
- 全局 Provider

特点：

- 不承载具体业务决策
- 只负责组织结构和生命周期

### 3.2 Domain 页面组件

职责：

- 各业务包的页面入口
- 页面级交互编排
- 与服务层通信

特点：

- 可组合多个面板
- 不直接承担底层协议实现

### 3.3 Workspace / Panel 组件

职责：

- 左侧资源栏
- 中间画布 / 播放器 / 编辑区
- 右侧属性面板
- 顶部工具栏
- 底部状态栏

这类组件是创作类产品的核心复用层，应跨 `film`、`magic-cut`、`canvas` 形成统一的布局语言。

### 3.4 Primitive / Design 组件

职责：

- Button、Dialog、Tabs、Tooltip、Popover、Form 控件
- 面板容器
- 标准列表
- 资源卡片

当前技术栈已使用 `Radix UI`、`Tailwind` 等基础能力，适合作为交互一致性的底层支撑。

## 4. 状态管理设计

### 4.1 当前状态特征

系统当前已呈现出多种状态类型共存的特点：

- UI 短状态：面板开关、选择态、分页、筛选
- 业务状态：工作区、项目、资产、聊天、生成任务
- 引擎状态：时间线、节点图、镜头结构、预览状态
- 平台状态：会话、环境、更新、运行时能力

### 4.2 推荐状态分层

| 状态类型 | 推荐存放位置 |
| --- | --- |
| 页面局部状态 | 页面 / 容器组件局部 state |
| 领域状态 | 领域 store 或 service facade |
| 长生命周期编辑状态 | 引擎专属 store / session model |
| 持久化偏好 | settings / runtime storage |
| 远端资源状态 | SDK service + query model |

关键原则：

- 不把所有状态塞到全局 store
- 不把所有业务状态塞到页面组件
- 不让引擎状态和普通页面状态混在一起

## 5. 组件化设计标准

### 5.1 统一命名与职责

建议统一约定：

- `Page` 结尾：路由级页面
- `Panel` 结尾：工作区面板
- `Toolbar` 结尾：工具条
- `Sidebar` 结尾：侧栏
- `Dialog` / `Sheet`：模态交互
- `Service`：业务服务
- `Facade` / `BusinessService`：领域统一入口

### 5.2 复用标准

若某组件满足以下条件，应晋升为共享组件：

1. 在两个以上模式复用。
2. 具备稳定输入输出，不依赖单一页面上下文。
3. 拆出后可以降低页面复杂度。

不应复用的情况：

- 仅在单一引擎中强绑定领域语义的组件
- 为了“看起来通用”而过度抽象的组件

## 6. 当前前端架构风险

| 风险 | 描述 | 影响 |
| --- | --- | --- |
| 多模式 UI 语言可能分裂 | `portal`、`film`、`magic-cut`、`canvas` 风格和交互可能各自演化 | 体验碎片化 |
| 复杂引擎状态与页面状态混杂 | 长期会增加维护成本 | 难以测试和扩展 |
| 业务服务旁路风险 | 个别页面可能直接接入特定能力 | 边界退化 |
| 对话式编辑缺少动作层 | Chat 结果还未普遍转成可执行编辑命令 | AI 价值难沉淀 |

## 7. 行业领先前端标准

要达到行业领先水平，前端必须具备：

1. 统一的工作台视觉与交互语言。
2. 面板、播放器、资源卡、属性区等高度复用。
3. 引擎级页面具备独立的状态模型和性能优化策略。
4. 可插拔面板与可配置工作区布局能力。
5. AI 结果可以自然转成页面动作、属性修改和工程操作。

## 8. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| App Shell 稳定性 | 新功能接入不破坏主壳结构 | `L4` | 继续坚持路由注册和布局分层 |
| 组件复用度 | 核心面板和基础控件高复用 | `L3-L4` | 统一工作区组件语言 |
| 状态可维护性 | 引擎状态、业务状态、UI 状态清晰分层 | `L3` | 建立引擎 session / store 标准 |
| 体验一致性 | 多模式视觉与交互统一 | `L3` | 梳理跨模式设计标准 |
| 可测试性 | 复杂页面核心逻辑可独立测试 | `L3` | 抽离业务逻辑到服务和 store |

## 9. 结论

Magic Studio V2 的前端已经具备桌面工作台雏形，但要达到行业领先标准，必须继续从“页面集合”升级为“统一工作台系统”，尤其要在引擎状态模型、面板复用、AI 动作化和多模式一致性上继续深化。
