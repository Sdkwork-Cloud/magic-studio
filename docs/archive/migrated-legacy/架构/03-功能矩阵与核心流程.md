# 03. 功能矩阵与核心流程

## 1. 功能矩阵

### 1.1 一级能力矩阵

| 一级能力 | 代表模块 | 当前状态 | 说明 |
| --- | --- | --- | --- |
| 创作入口 | `portal-video` | 较成熟 | 承接多模态输入、风格与参数选择 |
| 图像生成 | `image` | 较成熟 | 适合作为视觉资产生产入口 |
| 视频生成 | `video` | 较成熟 | 支持文生视频、图生视频、风格迁移、延长、口型同步 |
| 音频生成 | `music` `sfx` `voice` `audio` | 成熟中 | 多条声音能力线并存 |
| 影视化创作 | `film` | 成熟中 | 具备从剧本到分镜的结构化编辑方向 |
| 时间线剪辑 | `magiccut` | 成熟中 | 已具备专业工作区形态 |
| 节点式创作 | `canvas` | 成熟中 | 已具备复杂画布交互能力 |
| 对话式编辑 | `chat` `prompt` | 较成熟 | 可作为 AI 编辑和提示词优化入口 |
| 工程管理 | `workspace` `project` `assets` | 较成熟 | 已具备平台底座特征 |
| 平台扩展 | `skills` `plugins` | 技术预备期 | 技能较成熟，插件运行时不足 |

### 1.2 按创作阶段看能力覆盖

| 创作阶段 | 已有能力 | 仍需强化 |
| --- | --- | --- |
| 灵感输入 | Portal、Chat、Prompt、脚本上传、素材导入 | 统一意图理解与任务编排 |
| 素材生产 | 图像、视频、音乐、语音、角色生成 | 模型治理、结果评分与自动归档 |
| 工程组织 | Workspace、Project、Asset Center、Notes | 跨模式统一项目图谱 |
| 结构化创作 | Film、Canvas、MagicCut | 模式间引用与可转换性 |
| 预览与编辑 | 时间线、画布、影视编辑工作区 | 统一历史、撤销重做、协同协议 |
| 导出与发布 | 导出入口存在 | 工程同步、发布、渠道投放仍需补齐 |

## 2. 核心业务流程

### 2.1 统一创作入口流程

```text
用户输入需求
  -> 选择模型 / 风格 / 时长 / 比例 / 素材
  -> 创建创作会话
  -> 路由到 image / video / film / music / character 等模式
  -> 生成结果进入资产中心
  -> 可继续进入 canvas / magic-cut / film 等结构化模式
```

该流程是产品的“总入口”，目标不是只负责跳转，而是负责形成可追踪的创作上下文。

### 2.2 资产导入与统一归档流程

```text
导入本地文件 / 生成结果 / 外部链接
  -> AssetCenterService.importAsset / registerExistingAsset
  -> 媒体分析、定位、存储模式判断
  -> 生成统一资产记录
  -> 绑定 workspace / project / domain reference
  -> 在各创作模式中按 assetId 引用
```

该流程是平台化的关键。只要绕开资产中心，后续项目治理、引用追踪、清理回收都会失控。

### 2.3 视频生成任务流程

```text
提交视频生成请求
  -> 生成 API / SDK 创建异步任务
  -> 前端轮询任务状态
  -> 输出资源归档
  -> 绑定项目 / 资产引用
  -> 进入后续编辑或导出
```

当前视频服务已支持多种任务类型，但还需要统一任务状态、成本、取消、重试与审计标准。

### 2.4 Film 创作流程

```text
脚本输入或上传
  -> 创建 film project
  -> 编辑角色 / 场景 / 道具 / 分镜
  -> AI Copilot 辅助生成镜头或内容
  -> 结果沉淀为项目资源
  -> 预览、编辑、导出或进入剪辑
```

Film 模式的价值在于“结构化叙事创作”，而不是简单把视频生成功能换个页面包装。

### 2.5 MagicCut 剪辑流程

```text
创建或打开工程
  -> 导入资产 / 绑定资源
  -> 时间线编排
  -> 预览播放器
  -> 属性面板调整
  -> 模板保存 / 导出
```

当前结构已经具备剪辑工作台基本形态，但仍需继续补强性能预算、历史记录、导出编排与插件化效果链。

### 2.6 Canvas 节点式创作流程

```text
创建画布
  -> 拖入节点 / 素材 / 生成能力
  -> 建立连接与分组
  -> 调整参数与布局
  -> 执行或输出中间结果
  -> 回写资产或项目
```

Canvas 的价值是承接高级工作流和复杂编排，需要与资产、AI、工程上下文强绑定，而不是成为独立小应用。

### 2.7 对话式编辑流程

```text
用户自然语言输入
  -> Chat / Prompt 服务解析意图
  -> 调用模型增强、生成或建议
  -> 将结果转为可执行编辑动作或提示词结果
  -> 输出到资产、项目或当前编辑上下文
```

当前对话能力更偏辅助工具，下一阶段应朝“可执行编辑代理”方向演进。

## 3. 关键流程上的现状判断

| 流程 | 当前成熟度 | 主要问题 |
| --- | --- | --- |
| 入口到生成 | `L4` | 参数和上下文可进一步统一抽象 |
| 生成到资产 | `L4` | 多模态归档标准还需全面覆盖 |
| 资产到工程 | `L3-L4` | 引擎间共享协议需加强 |
| 影视创作闭环 | `L3` | 需继续深化项目结构和生成一致性 |
| 剪辑闭环 | `L3-L4` | 需强化导出、模板、性能和插件效果链 |
| 画布闭环 | `L3` | 执行模型和工作流治理需加强 |
| 工程发布闭环 | `L2` | 同步与发布仍未产品化 |

## 4. 行业领先功能设计标准

要达到行业领先水平，每条核心流程都应满足以下标准：

1. 可回到项目上下文，不产生孤儿结果。
2. 可追踪资产来源、处理链路、模型参数和生成历史。
3. 支持撤销、重试、替换、比较和二次编辑。
4. 大任务具备异步执行、进度反馈、失败恢复和取消能力。
5. 同一用户可在“轻量入口”和“专业模式”之间无损切换。

## 5. 本章评估标准

| 评估项 | 行业领先标准 | 当前判断 | 达标动作 |
| --- | --- | --- | --- |
| 功能链路闭环 | 从输入到发布完整闭环 | `L3` | 实现真实发布和工程同步 |
| 模式切换一致性 | 入口、影视、剪辑、画布之间可无损切换 | `L3` | 建立统一项目图谱 |
| 任务治理 | 所有长任务有状态、进度、取消、审计 | `L3-L4` | 统一任务协议 |
| 资产归档率 | 生成结果默认归档、可追踪 | `L4` | 禁止旁路落盘 |
| 可运营性 | 关键流程可度量、可优化 | `L2-L3` | 建立全链路观测指标 |

## 6. 结论

Magic Studio V2 已具备非常强的功能广度，下一步必须将功能矩阵提升为“流程矩阵”。只有当主要流程真正闭环，广度才会转化为产品竞争力，而不是模块堆叠。
