# Step并行车道交付包与集成交接标准

## 1. 文档定位

本文件规定并行 lane 在实现、验证、合流和阻塞升级时，必须提交什么样的交付包，才能安全进入集成阶段。

目标不是“代码写了就能交接”，而是：

`只有边界冻结、证据齐备、回滚清晰、下游可接，才允许交接。`

## 2. 合流前的强制前置项

每个 lane 在准备交接前，必须同时满足：

1. 当前 step 的开工卡与收尾卡已填写完成。
2. 当前 lane 本地验证已经完成。
3. 当前 lane 已产出 `docs/review/` 证据。
4. 当前 lane 已准备能力兑现证明包。
5. 当前 lane 已明确本轮输入、输出、变更目录、共享文件触达与回滚方式。

缺任一项，不允许进入集成窗口。

## 3. 边界冻结单

每个 lane 合流前必须附带一份边界冻结单，说明：

- 当前 lane 的 base commit SHA
- 当前 lane 的主写目录
- 当前 lane 触达的共享文件
- 本轮新增或变更的协议
- 禁止下游在接手前做的事情

## 4. 每个 Lane 必交内容

| 类别 | 必交内容 |
| --- | --- |
| 输入输出 | 本轮输入、非目标、最小输出、当前阻塞等级 |
| 代码范围 | 变更文件清单、主写目录、共享文件触达说明 |
| 验证证据 | must-run 命令、结果摘要、失败路径摘要 |
| 风险治理 | 当前风险、回滚方式、不可逆变更说明 |
| 交接约束 | 下游依赖、禁止动作、冻结协议版本 |
| 文档回写 | `docs/review/` 路径、`docs/架构/` 回写项、`docs/step/` 回写项 |

## 5. 集成交接模板

```md
## Lane 名称
## 当前 Step
## base commit SHA
## 当前输入
## 本步非目标
## 当前最小输出
## 变更目录
## 变更文件清单
## 共享文件是否触达
## 冻结的接口 / 协议
## must-run 命令
## 结果摘要
## 当前阻塞等级
## docs/review 证据路径
## docs/架构 回写项
## 风险与回滚
## 下游依赖
## 当前是否允许合流
```

## 6. 集成窗口检查项

集成 owner 在合流前必须逐项确认：

1. 当前 lane 是否有未关闭阻塞。
2. 当前 lane 是否抢改了共享文件。
3. 当前 lane 的 review 证据是否齐备。
4. 当前 lane 的 must-run 命令是否全部通过。
5. 当前 lane 是否明确说明了回滚方式。
6. 当前 lane 是否已经给出下游依赖和禁止动作。

## 7. 与 `92` 的联动升级

一旦交接时发现以下问题，必须按 `92` 升级：

- 本轮输入不清
- 冻结协议不清
- 当前阻塞等级不清
- 需要谁决策不清
- 当前是否允许继续推进不清

结论：

- 信息不清，不是“小问题”
- 信息不清，默认阻塞

## 8. 不允许交接的状态

出现以下任一情况，当前 lane 交接直接驳回：

1. 没有本地验证结果。
2. 没有 `docs/review/` 证据。
3. 没有回滚说明。
4. 共享文件被多个 lane 同时主改。
5. 冻结协议未写明。
6. 只提交代码，不提交风险、验证和交接信息。

## 9. 结论

并行 lane 的真正交接标准，不是“分支能 merge”，而是：

- 有边界冻结单
- 有验证证据
- 有 review 产物
- 有回滚说明
- 有下游可执行的交接信息

否则，这个 lane 只能叫“实现进行中”，不能叫“可集成交付”。
