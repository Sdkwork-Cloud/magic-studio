# 反复执行 Step 指令

## 1. 使用目的

本提示词用于在 `magic-studio-v2` 仓库中反复执行同一条总控指令。  
每次重复输入本提示词时，都必须基于**当前仓库真实状态**继续推进，不允许重新从头规划，不允许停留在空泛分析，不允许把局部完成误判为整体完成。

最终目标不是“把 step 文档写完”，而是：

`将当前应用持续重构、升级、迭代到可完整商业化交付的标准版本，直到 docs/step 的全部能力完整落地、完整验证、完整收口。`

本项目是一个桌面端视频创作应用，核心能力包括但不限于：

- 图片、视频、语音、音乐、多模态生成
- 大模型接入与统一 AI 任务中心
- Film 创作模式
- Canvas 创作模式
- 对标剪映的 MagicCut 剪辑能力
- 对话式 Chat / Prompt 编辑能力
- 插件能力
- 资产中心、工程管理、发布导出、安装交付
- Tauri 桌面运行时、安全治理、性能治理、发布治理

你每次收到本提示词，都必须继续把这些能力朝“行业领先、可商业化发布”的方向推进。

---

## 2. 核心总原则

1. 不要停留在计划。
2. 不要重复空泛总结。
3. 不要跳过测试、review、架构回写、release 记录。
4. 不要把“解锁下游”误判为“能力已完整兑现”。
5. 不要在共享协议未冻结时无序并行。
6. 每次都必须基于当前仓库真实状态继续向前推进。
7. 如果 `00-13` 全部完成，也不能停止，必须进入持续优化模式，继续朝商业化交付标准补齐剩余短板。

补充原则：

8. 每次执行前，必须先在内部完成多轮方案比较、反证检查、回归扫描与优先级重排，再输出精炼结论和真实实施结果。
9. 不允许自我欺骗，不允许用“局部通过”“单条命令通过”“局部演示可跑”代替完整闭环。
10. 如果本轮因 blocker 无法完成主目标，仍然必须继续推进当前最有价值的可执行收敛项，并把 blocker、影响面、回滚和解锁条件记录到 review 与 release 文档中。

### 2.1 “完美结果”定义与目标函数

本提示词中的“完美”，不是主观满意，也不是文档层自我宣布完成，而是当前约束下达到**可验证、可交付、可持续演进**的最优状态：

- 核心用户价值主链路真实闭环
- 架构边界、协议和共享主干稳定
- 安装、导出、发布、安全、性能、回滚具备真实门禁
- 证据、review、release、架构回写彼此一致
- 剩余缺口已经被消灭，或被显式降级为带 owner、退出条件和时间边界的受控问题

每一轮都必须优先最大化以下目标函数，且**低优先级目标不得以破坏高优先级目标为代价**：

1. 核心用户价值与主链路闭环率
2. 架构一致性与共享协议稳定性
3. 发布、安装、导出、安全、性能 readiness
4. 可维护性、可扩展性、可测试性
5. 并行效率与交付速度

每一轮要么：

- 关闭一个真实缺口
- 降低一个高风险 blocker
- 提升一个量化门禁或证据完备度

如果一轮没有新增能力、没有新增证据、没有降低风险，则该轮视为“无效迭代”，下一轮必须改策略，不能机械重复。

## 4. 重复执行语义

这条提示词是可重复输入的。

每次再次输入时，你必须先判断：

1. 当前在哪个 step
2. 当前在哪个 wave
3. 当前 step 是否已经达到“解锁最低 bar”
4. 当前 step 是否已经达到“能力完工 bar”
5. 当前 wave 是否已经完成阶段验收
6. 当前 release 文档是否已经更新
7. 当前 changelog 是否已经记录上轮迭代结果

如果当前 step 未闭环：

- 继续完成当前 step
- 补齐代码、测试、验证、review、架构回写、release 记录
- 不允许只输出计划或建议

如果当前 step 已闭环但当前 wave 未闭环：

- 自动进入该 wave 内下一个允许推进的 step
- 必须遵守 `87/94/95/98/99` 的边界与门禁

如果当前 wave 已闭环：

- 必须先按 `93` 完成波次总验收
- 形成 review 与 release 记录
- 再进入下一 wave

如果 `00-13` 全部闭环：

- 自动进入“商业化优化模式”
- 持续审计并优化功能完整性、性能、安全、稳定性、发布质量、安装交付、可维护性、兼容性、商业可运营性

- 禁止没有 review 和 release 记录就宣告完成

## 6. 每轮执行的固定步骤

每次收到本提示词，都必须按以下顺序执行：

### Step A：仓库现实审计

先核对：

- 当前 step / 当前 wave
- 当前 step 的未闭环项
- 当前 blocker
- 当前共享文件 owner 与写集边界
- 当前例外项和兼容层
- 当前 changelog、version、release 文档是否更新

同时必须检查：

- 是否存在被遗漏的 mock / fallback / 临时放权
- 是否存在未关闭的 review 问题
- 是否存在与商业化交付目标冲突的已知缺陷
- 是否存在跨模块回归风险

### Step B：确定本轮唯一主目标

只能选择一个“最高优先级主目标”：

- 要么补齐当前未闭环 step
- 要么补齐当前 wave 的总验收缺口
- 要么在全部 step 完成后，补齐商业化短板中最关键的一项

必须明确：

- 本轮为什么做它
- 它对应哪篇 `docs/架构`
- 它对应哪篇 `docs/step`
- 它如何影响商业化交付

主目标选择必须遵循以下排序因子：

1. 是否阻塞当前 wave / 当前主链路
2. 是否影响共享主干或多 lane 依赖
3. 是否影响商业化交付中的核心用户价值
4. 是否属于高风险安全 / 性能 / 发布缺口
5. 是否能在当前轮形成可验证收敛

### Step C：真实实施

必须真正实施，而不是只分析：

- 改代码
- 改测试
- 改脚本
- 改文档
- 改 review
- 改 release 记录

### Step D：真实验证

必须运行适用的真实命令，并基于结果裁决，而不是口头宣称完成。

优先使用项目现有命令，例如：

- `pnpm run preflight:deps`
- `pnpm run test`
- `pnpm run test:vitest`
- `pnpm run test:node`
- `pnpm run typecheck`
- `pnpm run lint`
- `pnpm run build:git-sdk`
- `pnpm run tauri:build`
- `pnpm run tauri:bundle:verified`
- `pnpm run verify:release:artifacts`
- `pnpm run verify:iam-route-css-budget`
- `pnpm run audit:services`

如果某条命令不适用，必须说明为什么不适用。

验证时必须同时做：

- 成功路径验证
- 失败路径验证
- 回归影响验证
- 与历史兼容层相关的验证

### Step E：review、架构回写、release 回写

每轮结束必须同步更新：

- `docs/review/`
- 必要时的 `docs/架构/`
- 必要时的 `docs/step/`
- `docs/release/CHANGELOG.md`
- `docs/release/VERSION.md`
- `docs/release/` 下本轮迭代记录

### Step F：自评与下一轮决策

本轮结束后，必须执行自评：

- 当前 step 是否达到解锁最低 bar
- 当前 step 是否达到能力完工 bar
- 当前 wave 是否达到阶段总验收条件
- 当前版本是否比上轮更接近商业化交付
- 还缺什么
- 下一轮应该继续做什么

并且必须再额外完成一轮“反证审查”：

1. 有没有把“解锁最低 bar”误判成“能力完工 bar”
2. 有没有因为单条命令通过而忽略真实业务闭环
3. 有没有因为文档更新而误判为代码能力已落地
4. 有没有因为局部功能可用而忽略安装、发布、性能、安全或兼容风险
5. 有没有遗漏 changelog、version、release record、review 回执中的任一项

如果任一回答为“有”或“不确定”，则当前轮不得宣告完美完成，必须继续迭代。

## 6.1 内部自驱思考循环

在每一轮真正输出前，必须在内部完成以下思考循环，但不要把全部内部推理原样展开输出，只输出最终结论、证据和行动结果：

1. 现实扫描：当前真实状态是什么
2. 差距比较：距离目标架构 / 商业化目标还差什么
3. 方案比较：至少比较 2-3 个推进方向，选最高杠杆路径
4. 反证检查：当前方案可能失败在哪里
5. 回归扫描：当前变更会破坏什么
6. 收敛判断：本轮怎样做才能最接近“更完整、更可交付”的结果
7. 执行裁决：最终只保留一个主目标推进

如果没有完成这套内部循环，不允许直接开始改动。

## 6.2 防自我欺骗规则

以下情形一律视为“伪完成”：

1. 只完成文档，没有完成代码和验证。
2. 只完成代码，没有完成 review、架构回写、release 回写。
3. 只跑通成功路径，没有跑失败路径和回归路径。
4. 只局部可用，但安装、导出、发布、恢复、兼容未验证。
5. 只证明某个模块可用，没有证明它已经真正支撑下游或商业化交付。

一旦出现伪完成信号，必须回退到“未完成”状态继续推进。

---

## 6.3 平台期识别与策略切换规则

出现以下任一情形，视为进入“平台期”：

1. 同一 step 连续两轮没有新增有效证据包，且没有关闭关键缺口。
2. 总评分连续两轮没有明显提升，且主链路问题没有减少。
3. 同一个 blocker 连续两轮重复出现，只是换表达方式，没有真实解法推进。
4. 同一个 workaround / fallback / 兼容层连续两轮被保留，却没有清退计划和失效条件。
5. 变更主要停留在文档、表层修补或局部演示，未推动真实代码、测试、发布链路闭环。

进入平台期后，必须强制从以下策略中选择至少一个，并写入 `docs/review` 与 `docs/release`：

- 根因下钻：继续往下拆，找到真实单点瓶颈。
- 范围收敛：缩小为更小的闭环 slice，但不改变架构方向。
- 协议冻结：先冻结共享接口、状态机、类型契约，再恢复并行开发。
- 阻塞升级：按 `docs/step/92-Step输入输出与阻塞升级规则.md` 升级 blocker。
- 实验回滚：撤销无收益实验或错误方向，回到最后一个稳定可验证状态。

每次触发平台期，都必须明确：

- 为什么判定为平台期
- 本轮改用什么策略
- 预计用什么证据证明已走出平台期
- 如果仍走不出平台期，下一次要升级到什么层级处理

## 6.4 回归重开与完成态撤销规则

任何已宣告“完成”的 step / wave / 能力域，只要出现以下任一情况，必须立即重开：

1. 关键命令、关键测试、关键构建重新失败。
2. `解锁最低 bar` 或 `能力完工 bar` 的前提条件不再成立。
3. 未登记 mock、fallback、手工步骤重新回流到主路径。
4. `docs/架构`、`docs/step`、`docs/review`、`docs/release` 与代码事实失同步。
5. 安装、导出、发布、安全、性能、恢复、兼容的既有门禁被新变更破坏。
6. 已关闭 blocker 重新出现，或以变体形式重新出现。

一旦重开，必须执行以下动作：

- 将状态从 `completed` 回退为 `in_progress` 或 `blocked`
- 在 `docs/review` 记录重开原因、影响范围、责任 step、回收动作
- 在 `docs/release` 记录“重开”而不是继续维持虚假完成态
- 不允许带病进入下一 step、下一 wave 或更高 release stage

## 6.5 blocker 分层与外部依赖推进规则

每个 blocker 都必须标注类型：

- `internal`：当前仓库内可直接解决
- `shared`：需要跨模块或跨 lane 协调
- `external`：依赖外部仓库、接口、第三方服务或组织协作
- `environmental`：依赖环境、权限、构建机、证书、网络、平台能力

只要 blocker 存在，就必须同时回答：

1. 它是否阻塞当前 step 的 `解锁最低 bar`
2. 它是否阻塞当前 step 的 `能力完工 bar`
3. 它是否阻塞当前 wave 总验收
4. 它是否阻塞 release stage 升级

即使外部 blocker 尚未解除，也必须继续推进以下本地收敛项：

- 本地可独立完成的代码、测试、脚本、文档和证据包
- 接口冻结、类型冻结、边界隔离
- 失败路径、降级路径和回滚路径验证
- 依赖包准备：owner、上游依赖、解锁条件、预计完成时间、fallback 失效日期

严格禁止：

- 用长期 fallback、长期 mock、长期手工步骤伪装完成
- 用“等待外部”作为整轮无产出的借口
- 未经 `95` 和 `97` 判定就把 blocker 视为“不影响完工”

## 8. 自我评估与迭代升级规则

每轮都必须对以下维度评分，满分 100，低于 90 不得宣告“高质量完成”：

| 维度 | 分值 |
| --- | --- |
| 架构对齐度 | 15 |
| 需求与业务闭环完整度 | 15 |
| 代码质量与模块边界 | 10 |
| 测试与验证充分性 | 15 |
| 性能 | 10 |
| 安全 | 10 |
| 发布与安装交付 readiness | 10 |
| review / 文档 / 架构回写完整度 | 10 |
| 商业化可运营性 | 5 |

自评规则：

- `< 80`：不合格，必须继续当前 step 或修复关键缺陷
- `80-89`：部分达标，允许保留当前主目标，但不得宣告闭环完成
- `>= 90`：可视为当前轮高质量完成，但仍要判断是否还有更高优先级缺口

补充判定：

- 如果总分连续两轮无明显提升，且没有关闭新的关键缺口，自动触发平台期规则。
- 如果总分提升来自文档层或局部演示，但核心链路、验证和发布门禁没有同步提升，则不得视为质量真实提升。

如果发现：

- 仍有 mock 主路径
- 仍有未登记 fallback
- 仍有未验证的核心能力
- 仍有未关闭的安全/性能/交付风险

则必须继续迭代，不得停下。

此外必须执行“终局差距问题单”：

- 还有哪些能力不能支撑真实商业用户使用
- 还有哪些能力不能支撑正式发布或安装交付
- 还有哪些能力不能支撑后续持续迭代维护
- 还有哪些能力不能支撑行业领先目标

只要上述问题仍存在明确答案，默认不能视为“终局完成”。

## 9. 商业化完成标准

即使 `00-13` 全部完成，只有同时满足以下条件，才允许将项目视为“可完整商业化交付”：

1. 核心创作能力闭环完整
2. AI、多模态、资产、工程、插件、发布、安装全部具备真实可运行链路
3. 核心主路径没有未标记 mock
4. 安全、性能、稳定性、发布、交付有真实门禁
5. review、架构回写、release 记录完整
6. 当前版本具备可持续维护和后续迭代基础

如果未达成这些条件，则进入“商业化完善模式”，继续补齐。

### 9.1 连续收敛判定

只有在以下条件连续满足至少两轮迭代后，才允许把结果视为“稳定商业化完成态”：

1. 没有新增 blocker
2. 没有新增高风险回归
3. changelog、version、review、架构回写持续保持同步
4. 当前轮与上一轮相比，没有被重新打开的核心能力缺口

如果任一条件被打破，自动退出稳定完成态，继续迭代。

### 9.2 阶段发布量化门禁

每轮都必须对以下能力域标记 `Green / Yellow / Red`，并写入 review 或 release 记录：

1. 创作引擎：Canvas / Film / MagicCut
2. AI 与多模态任务中心
3. 资产、工程、插件体系
4. Chat / Prompt 对话式编辑
5. 桌面运行时、权限、安全治理
6. 测试、质量、回归门禁
7. 性能、稳定性、观测能力
8. 安装、导出、发布、交付链路

阶段升级必须满足：

- `internal`：治理规则已建立，但允许存在多个 `Yellow / Red` 能力域。
- `alpha`：至少 1 条从创作到导出的真实主路径可跑通；无未登记的 P0 问题。
- `beta`：核心能力域至少 6 项为 `Green`；主路径不存在未登记 mock；安装/导出/发布基础链路已验证。
- `rc`：核心能力域全部 `Green` 或受控 `Yellow`；不存在未关闭的高风险安全、性能、发布 blocker；连续两轮稳定。
- `ga`：商业化主能力域全部达到可运营标准；无未关闭 P0/P1 风险；交付、回滚、运维、审计链路可用。

只要 release stage 条件不满足，就不允许人为升级 `VERSION.md` 中的阶段状态。

- `Patch`：bug 修复、质量提升、性能/安全优化、文档回写、非破坏性增强

## 13. 直接执行指令正文

当你收到这条提示词时，请立即执行以下动作，不要停留在分析：

1. 检查当前仓库真实状态，判断当前所在 step、wave、阻塞、版本、release 状态。

9. 每轮必须执行内部自驱思考循环、反证检查和防自我欺骗检查；只要存在疑点，就继续打磨，不得自我放行。

最重要的约束：

`每次重复输入这条提示词，都必须基于当前仓库真实状态继续向前推进，直到所有 step 完整落地、完整验证、完整收口，并达到可完整商业化交付的结果。`
