# Step 13 - 集成联调验收与正式发布门禁

## 1. 目标与范围

本 step 用于把 `Step 00-12` 的产物汇总为统一验收、统一发布结论和持续迭代入口，确保架构能力不是名义完成，而是通过统一证据证明已实现。

### 1.1 执行输入

- `docs/架构/13-测试质量-安装部署-发布运维.md`
- `docs/架构/14-行业对标-评估标准与演进路线.md`
- `docs/架构/15-架构落地步骤-并行实施计划.md`
- `Step 06-12` 的所有验证结果和交付包

### 1.2 本步非目标

- 不在本 step 内继续修改共享协议
- 不在本 step 内用临时补丁掩盖前序缺陷

### 1.3 最小输出

- E2E 验收矩阵
- 发布就绪清单
- Go / No-Go 结论
- 下一轮 backlog 入口

## 2. 架构对齐

- `docs/架构/01-15`
- 尤其对齐 `docs/架构/15` 的 `Step 12`

## 3. 当前现状

当前已有较多测试脚本与构建脚本，但仍需要统一把：

- Film
- MagicCut
- Canvas
- Chat / Prompt
- 插件
- 导出 / 发布
- 性能预算

纳入同一张验收矩阵。

## 4. 设计

### 4.1 E2E 主链路

至少覆盖：

- `Portal -> 生成 -> 资产`
- `Film -> 分镜 -> 生成 -> 预览`
- `MagicCut -> 导入 -> 编辑 -> 导出`
- `Canvas -> 节点执行 -> 资产 / 时间线输出`
- `Chat / Prompt -> 动作 -> 引擎执行`
- `插件安装 -> 权限申请 -> 运行`

### 4.2 发布结论

必须输出三类结论之一：

- `可发布`
- `有条件可发布`
- `不可发布`

## 5. 实施落地规划

1. 汇总各 lane 交付包和验证结果
2. 按 E2E 矩阵跑关键链路
3. 对照 `docs/架构/01-15` 做能力回写校验
4. 输出发布就绪清单和下一轮 backlog

## 6. 测试计划

```bash
pnpm typecheck
pnpm test
pnpm audit:services:policy
pnpm verify:sdk-mode-parity
pnpm verify:mode-style-parity
pnpm verify:release:artifacts
pnpm tauri:build
```

同时必须执行人工或半自动 E2E 主链路验证。

## 7. 结果验证

预期结果：

- 不存在阻断发布的架构旁路和关键回归
- 可明确给出 Go / No-Go
- 所有关键链路都有证据

## 8. 检查点

- `CP13-1`：主链路验收矩阵跑通
- `CP13-2`：统一门禁通过
- `CP13-3`：架构回写完成
- `CP13-4`：发布结论与下一轮 backlog 输出完成

## 9. 推荐并行执行

- step 级：必须串行，最后执行
- 子任务级：E2E 执行、证据汇总、架构回写可由 QA owner、架构 owner、集成 owner 并行

## 10. 风险与回滚

风险：

- 验收阶段暴露跨 lane 隐性问题
- 证据不完整导致无法给出发布结论

回滚：

- 验收缺口回到对应 lane 修复
- 不在本 step 内堆叠临时修补

## 11. 完成定义

- 能证明架构文档中的核心能力已经被完整实现
- 发布依据、残余风险和下一轮演进方向明确

## 12. 下一步准入条件

- 仅在 `Go` 或 `Conditional Go` 条件下进入正式发布或下一轮迭代
